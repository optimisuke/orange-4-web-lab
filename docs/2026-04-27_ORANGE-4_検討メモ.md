## 概要

### ORANGE-4とは

**ORANGE-4**は、GMC-4互換の4ビットマイコン組立てキットです。プログラミング教育や組み込みシステムの学習に適した教材として設計されています。

**製品情報**:

- メーカー: ピコソフト
- 公式サイト: http://www.picosoft.co.jp/ORANGE-4/
- 価格: 約3,000円
- 特徴: GMC-4互換 + 拡張機能

**関連リンク**:

- ピコソフト公式: http://www.picosoft.co.jp/
- GMC-4（学研）: https://otonanokagaku.net/products/computer/gmc4/

### できること・できないこと

**✅ できること**:

- プログラミング学習（機械語・アセンブリ）
- LED制御（7セグメント + 7つのLED）
- 音声出力（ビープ音・メロディ）
- キー入力処理
- 簡単なゲーム開発
- PC連携（シリアル通信）

**❌ できないこと**:

- 複雑なアプリケーション開発
- グラフィック表示
- ネットワーク通信
- 大量のデータ処理

## 基本仕様

### ハードウェア仕様

- **CPU**: 4ビットマイコン（GMC-4互換）
- **メモリ**: 256バイト（GMC-4の3倍以上）
- **命令セット**: 基本16種類 + 拡張命令
- **I/O**: 7セグLED、7つのLED、キー入力、音出力

### メモリマップ

```
アドレス範囲  用途                    容量
00-4F        プログラム領域          80バイト
50-5F        データ領域              16バイト
60-7F        システム領域            32バイト
80-EF        プログラム/スタック      112バイト
F0-FF        スタック領域            16バイト
```

**実用容量**:

- プログラム: 150-170バイト
- データ: 16バイト
- スタック: 20-40バイト

### 命令セット

基本命令（GMC-4互換）:

| コード | ニーモニック | 説明                 |
| ------ | ------------ | -------------------- |
| 0      | KA           | キー入力             |
| 1      | AO           | A→7セグLED           |
| 2      | CH           | A↔B, Y↔Z             |
| 3      | CY           | A↔Y                  |
| 4      | AM           | A→M[Y]               |
| 5      | MA           | M[Y]→A               |
| 6      | M+           | A+M[Y]→A             |
| 7      | M-           | A-M[Y]→A             |
| 8      | TIA n        | n→A                  |
| 9      | AIA n        | A+n→A                |
| A      | TIY n        | n→Y                  |
| B      | AIY n        | Y+n→Y                |
| C      | CIA n        | A≠n?                 |
| D      | CIY n        | Y≠n?                 |
| E      | CAL n        | サブルーチン呼び出し |
| F      | JUMP nn      | ジャンプ             |

拡張命令の詳細は公式ドキュメントを参照してください。

## 用語解説

### クロスアセンブラとは

**クロス（Cross）の意味**: 開発環境と実行環境が異なること

#### 通常のアセンブラ

同じコンピュータ上で開発と実行を行います。

```
┌─────────────────┐
│  同じコンピュータ │
│                 │
│  アセンブラ実行  │
│      ↓          │
│  機械語生成      │
│      ↓          │
│  そのまま実行    │
└─────────────────┘
```

#### クロスアセンブラ

異なるコンピュータで開発と実行を行います。

```
┌─────────────────┐      ┌─────────────────┐
│  Mac/Windows    │      │  ORANGE-4       │
│  (開発環境)      │      │  (実行環境)      │
│                 │      │                 │
│  アセンブラ実行  │ USB  │                 │
│      ↓          │─────→│  機械語を実行    │
│  機械語生成      │転送  │                 │
└─────────────────┘      └─────────────────┘
   x86/ARM                  PIC 4bit
```

**ポイント**:

- Mac/Windowsで開発（x86/ARM CPU）
- ORANGE-4で実行（PIC 4bit CPU）
- アーキテクチャが異なるため「クロス」

### モニターコマンド形式（e00:）

ORANGE-4にプログラムを書き込む際、2つの形式があります。

#### 形式A: 純粋な機械語（バイナリ）

```
83 A0 4 82 6 1 F06
```

- CPUが直接実行する形式
- 人間には読みにくい
- ファイルに保存するとバイナリファイル

#### 形式B: モニターコマンド形式（テキスト）

```
e00:83A04826F06
```

- モニタープログラムへの命令
- 人間が読める
- テキストとしてコピペ可能

**`e00:` の構造**:

```
e     → editコマンド（書き込み命令）
00    → 00番地から書き込み開始
:     → 区切り文字
83... → 書き込むデータ（機械語）
```

**なぜ必要？**

ORANGE-4のモニタープログラムに「00番地から83A04826F06を書き込んで」と指示するためです。`e00:`がないと、どこに何を書き込むか分かりません。

### コンパイラとアセンブラの違い

#### コンパイラ

**高級言語 → 機械語**（1対多の変換）

```c
int x = 3 + 2;
    ↓ コンパイル
0110 1011 0011 0010 ...（複数の機械語命令）
```

特徴:

- 1行が複数の機械語命令になる
- 最適化が行われる
- 人間の言葉に近い表現

#### アセンブラ

**アセンブリ言語 → 機械語**（1対1の変換）

```assembly
ldi 3
    ↓ アセンブル
83（1つの機械語命令）
```

特徴:

- 1行が1つの機械語命令
- ほぼ1対1対応
- CPUの命令に近い表現

### プログラムの階層構造

同じ処理を異なるレベルで表現した例:

#### レベル1: 高級言語（C言語）

```c
int result = 3 + 2;
printf("%d", result);
```

- 人間が読みやすい
- CPUは直接実行できない

#### レベル2: アセンブリ言語

```assembly
ldi 3       ; 3をセット
ldyi 0      ; Y=0
st          ; メモリに保存
ldi 2       ; 2をセット
add         ; 足し算
outn        ; 表示
```

- 人間が読める（英語っぽい）
- CPUは直接実行できない
- コメントが書ける

#### レベル3: 機械語（16進数）

```
83 A0 4 82 6 1
```

- 数字だけ
- CPUが直接実行できる
- 人間には読みにくい

#### レベル4: 2進数（CPUの内部）

```
10000011 10100000 00000100 10000010 00000110 00000001
```

- 電気信号のON/OFF
- CPUが実際に処理する形式

### 変換の流れ

```
高級言語（C等）
    ↓ コンパイル
アセンブリ言語
    ↓ アセンブル
機械語（16進数）
    ↓ 実際は
2進数（電気信号）
```

### 高級言語の実装

#### 関数ベース言語の提案

**コンセプト**: 演算子なし、関数呼び出しのみ

4ビットマイコンの制約を考慮し、演算子を使わず関数呼び出しのみで記述する言語を提案します。

#### 言語仕様例

```javascript
// 変数宣言と代入（=のみ）
let a = 5;
let b = 3;

// 演算は全て関数
let c = add(a, b); // a + b
let d = sub(a, b); // a - b

// 比較も関数
if (eq(a, 5)) {
  // a == 5
  display(a);
}

// ループも関数
repeat(5, function () {
  led_toggle(0);
  delay(100);
});
```

#### 組み込み関数一覧

**算術関数**:

```javascript
add(a, b); // a + b
sub(a, b); // a - b
mul(a, b); // a * b (遅い)
div(a, b); // a / b (遅い)
inc(a); // a + 1
dec(a); // a - 1
```

**比較関数**:

```javascript
eq(a, b); // a == b
ne(a, b); // a != b
gt(a, b); // a > b
lt(a, b); // a < b
```

**I/O関数**:

```javascript
display(n); // 7セグLED表示
input(); // キー入力
led_on(bit); // LED点灯
led_off(bit); // LED消灯
beep(note, ms); // 音出力
delay(ms); // 待機
```

#### 制約事項

1. **データ型**: 4ビット整数のみ（0-15）
2. **変数数**: 20-30個程度
3. **演算**: 加減算は高速、乗除算は低速
4. **制御構文**: 単純なif/while/for、ネスト浅い
5. **関数**: 引数1-2個、再帰不可
6. **メモリ**: 150-170バイト（実用）
7. **I/O**: 数値・LED・音のみ

#### 実装の見積もり

- **Phase 1: 基本機能**（8-10時間）
- **Phase 2: 制御構文**（3-4時間）
- **Phase 3: 拡張機能**（3-5時間）
- **Phase 4: 最適化**（2-3時間）
- **合計**: 16-22時間

#### 乗算・除算の実装

**実装方法**:

- **乗算**: 加算の繰り返し（`3 × 4 = 3 + 3 + 3 + 3`）
- **除算**: 減算の繰り返し（`12 ÷ 3 = 12 - 3 - 3 - 3 - 3`）

**パフォーマンス**:

- `mul(3, 4)`: 約24命令（実用的）
- `mul(10, 10)`: 約600命令（非常に遅い）
- `div(12, 3)`: 約32命令（実用的）
- `div(15, 1)`: 約120命令（遅い）

**推奨**:

- 小さい値（0-5）なら実用的
- 大きい値（10以上）は非推奨
- 2のべき乗版（`mul2`, `div2`）も提供

## ゲーム開発

### 作れるゲームの例

#### 1. 数当てゲーム

```javascript
let answer = random(16);
let tries = 0;

while_true(
  function () {
    return lt(tries, 5);
  },
  function () {
    let guess = input();
    tries = inc(tries);

    if (eq(guess, answer)) {
      beep_c5(500);
      return;
    }
  },
);
```

**特徴**:

- メモリ: 30-40バイト
- 難易度: ★☆☆☆☆

#### 2. 反射神経ゲーム

```javascript
let score = 0;

repeat(5, function () {
  delay(random(2000));
  let target = random(7);
  led_on(target);

  let pressed = input();
  led_off(target);

  if (eq(pressed, target)) {
    score = inc(score);
    beep_c4(100);
  }
});

display(score);
```

**特徴**:

- メモリ: 50-60バイト
- 難易度: ★★☆☆☆

#### 3. 2進数→16進数変換ゲーム ⭐おすすめ

```javascript
let score = 0;

repeat(5, function () {
  let answer = random(16);

  // 2進数でLED表示
  led_clear();
  if (bit_and(answer, 1)) led_on(0);
  if (bit_and(answer, 2)) led_on(1);
  if (bit_and(answer, 4)) led_on(2);
  if (bit_and(answer, 8)) led_on(3);

  let guess = input();

  if (eq(guess, answer)) {
    score = inc(score);
    beep_c5(200);
  } else {
    beep(40, 300);
  }
});

display(score);
```

**特徴**:

- メモリ: 50-60バイト
- 難易度: ★★☆☆☆
- 教育効果: ★★★★★

**おすすめの理由**:

- 7つのLEDで2進数を視覚的に表示
- 16進数で答えを入力
- プログラミングの基礎（2進数・16進数）を学べる
- 親子で楽しめる

#### 実践的なアドバイス

**1. キャリアステージ別の推奨**

**新人エンジニア（1-3年目）**:

- ❌ 低レイヤーは後回し
- ✅ 実務スキル優先（React/Python/AWS）
- ✅ ビジネス価値を生み出す力
- 理由: キャリアの基盤作りが最優先

**中堅エンジニア（4-7年目）**:

- ⚠️ 興味があれば学ぶ
- ✅ 専門性を深める
- ✅ アーキテクチャ設計力
- 理由: 差別化要因として有効

**シニアエンジニア（8年目以降）**:

- ✅ 教養として価値あり
- ✅ 技術的深みを持つ
- ✅ 後進の育成に活かす
- 理由: 技術的リーダーシップに必要

**2. 職種別の推奨**

| 職種                   | 低レイヤー学習 | 優先度 | 理由                 |
| ---------------------- | -------------- | ------ | -------------------- |

**3. 学ぶべきタイミング**

**学ぶべき時**:

- ✅ パフォーマンス問題に直面した時
- ✅ 技術的好奇心が強い時
- ✅ キャリアに余裕がある時
- ✅ 組み込み/ゲーム開発に転向する時

**学ばなくていい時**:

- ❌ 転職活動中
- ❌ 新しいフレームワーク習得中
- ❌ プロジェクトが忙しい時
- ❌ 実務スキルが不足している時

#### 結論: バランスが重要

**現実的な答え**:

1. **教養としては価値がある**
   - 技術的な深みを持てる
   - トラブルシューティング能力向上
   - 差別化要因になる

2. **でも必須ではない**
   - 大多数のエンジニアには不要
   - 実務で使う機会は少ない
   - 他のスキルの方が優先度高い

3. **興味と余裕があれば学ぶ**
   - 趣味として楽しむ
   - 長期的な投資として
   - キャリアの選択肢を広げる

**ORANGE-4の位置づけ**:

- 子供の教育: ★★★★★（最適）
- 組み込み志望: ★★★★★（必須）
- 教養として: ★★★☆☆（興味次第）
- キャリア重視: ★☆☆☆☆（優先度低）

**最終的なメッセージ**:

> 低レイヤー知識は「あると良い」が「必須ではない」。
> 自分のキャリアゴール、興味、時間的余裕を考慮して判断すべき。
> 無理に学ぶ必要はないが、興味があるなら学んで損はない。

## 付録

### よくある質問（FAQ）

#### Q1: ORANGE-4ってどんなことができる？

**A**: 4ビットマイコンで以下のことができます

- ✅ プログラミング学習（機械語・アセンブリ）
- ✅ LED制御（7セグ + 7つのLED）
- ✅ 音声出力（ビープ音・メロディ）
- ✅ キー入力処理
- ✅ 簡単なゲーム開発
- ✅ PC連携（シリアル通信）

#### Q2: WebAssemblyから機械語に変換できる？

**A**: 技術的には可能だが、非推奨

**理由**:

- ❌ アーキテクチャの根本的な違い（32/64ビット vs 4ビット）
- ❌ ほとんどのWATコードが変換不可能
- ❌ 実装コストが高い（30-50時間以上）
- ❌ 実用性が低い

**推奨代替案**:

- ✅ GMC-4専用アセンブラ（10-12時間）
- ✅ 簡易高級言語（15-20時間）
- ✅ マクロアセンブラ（5-8時間）

#### Q3: JavaScript風の高級言語は作れる？

**A**: 作れるが、大きな制約がある

制約事項については「実装ガイド > 高級言語の実装 > 制約事項」を参照してください。

#### Q4: ゲームは作れる？

**A**: シンプルなゲームなら作れます

詳細は「ゲーム開発」セクションを参照してください。

#### Q5: メモリは足りる？

**A**: 十分足ります

詳細は「基本仕様 > メモリマップ」を参照してください。

#### Q6: 低レベル知識はAI時代に必要？

**A**: 場合による。でも学んで損はない

詳細は「学習価値の評価」セクションを参照してください。

#### Q7: 結局、買うべき？

**A**: 目的次第

詳細は「学習価値の評価 > 推奨度」を参照してください。
